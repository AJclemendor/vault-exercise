# Vault Exercise

## What this is

This is a simplified off-chain matched, on-chain settled exchange. A `Vault` contract on a local Anvil chain acts as a settlement primitive — an operator calls `matchOrders(a, b, amountA, amountB)` to atomically swap tokens between two users. A simulation harness creates 200 users who continuously submit orders to an HTTP service and simultaneously perform chaotic on-chain activity (withdrawals, peer-to-peer transfers) that destabilize the service's view of the world.

Your job is to build that HTTP service in `service/`. It maintains an order book, validates incoming orders against on-chain balances, matches crossing orders, and settles them through the vault. The architecture is entirely yours.

The central challenge is **ghost orders** — orders that pass admission based on balance state that was true at submission time but is no longer true at settlement time. Users withdraw from the vault and transfer tokens to each other underneath your service. A user with 1000 tokens when you accept their order might have 400 by the time you try to settle. The exercise is fundamentally about designing a system that prevents or gracefully handles ghost orders while maintaining reasonable throughput.

We care far more about how you think about this problem than whether you achieve a 100% settlement success rate. A clean architecture that handles 70% of cases correctly and knows where it's weak is more valuable than a brittle hack that gets to 95%.

## What you've been given

- `contracts/` — Vault and MockToken (Solidity, Foundry)
- `harness/` — Rust simulation that creates load and chaos
- `scripts/` — `run.sh` orchestrates everything
- `config/local.json` — generated by setup, contains deployed addresses and the operator key your service needs

Read the contracts and harness source before writing code. The harness is your adversary. Understanding what it does is a prerequisite to building something that survives it.

## Setup and running

### Prerequisites

- Rust (via [rustup](https://rustup.rs/))
- Foundry (`curl -L https://foundry.paradigm.xyz | bash && foundryup`)
- jq (`brew install jq`)

```bash
./scripts/run.sh
```

## What to build in `service/`

An HTTP service on `localhost:8080`. The harness expects these endpoints:

**`POST /orders`** — Submit an order. Body: `{user, side, order_type, price, size}`. Returns `{order_id, status}` on success, 4xx on rejection.

**`DELETE /orders/:id`** — Cancel an open order.

**`GET /balances/:address`** — Balance breakdown: `{real, reserved, virtual_, vault}`.

**`GET /orders?user=:address`** — List open orders: `[{id, user, side, order_type, price, size, filled_size, status}]`.

`harness/src/service.rs` is the source of truth for exact JSON shapes, field names, and enum serialization. You are welcome to modify these functions however you like if necessary!

### Core responsibilities

- **Admission**: Validate that users have sufficient on-chain balance (minus existing reservations) to cover new orders. A certain percentage of incoming orders are intentionally oversized and must be rejected.
- **Order book + matching**: Price-time priority. Limit orders rest and market orders cross immediately.
- **Settlement**: Call `Vault.matchOrders(buyer, seller, buyerAmount, sellerAmount)` using the operator key from config. Handle reverts.
- **Balance reconciliation**: On-chain balances change constantly underneath you. Your service needs a strategy for keeping its view fresh enough to make good admission decisions without polling every user every tick.

### Required stats

Your service must track and expose operational stats so we can evaluate what it actually did under load:

- Orders received
- Orders accepted
- Orders rejected (admission failures)
- Orders matched
- Settlements attempted
- Settlements reverted
- Currently open orders

Periodically log these stats to the console, and include them in your final writeup.

### Design decisions you own

How you track balances (polling, events, caching, hybrid). How you handle settlement failures (retry, cancel, mark stale). How you manage concurrency across simultaneous orders for the same user. How aggressive your staleness tolerance is. How you batch or pipeline settlements. These are yours to decide and justify.

## `service.rs` and adapting it

`harness/src/service.rs` defines the exact request/response types the harness sends. Your service must be compatible with these shapes. If your API design genuinely requires a different shape, you can modify `service.rs` but ensure to document what you changed and why.
